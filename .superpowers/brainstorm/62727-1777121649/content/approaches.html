Manual Person Entry — 3 Approaches

You want name (multi-lang), phone, email, company, date, occasion, Met As — that's essentially the same fields as a scanned card.

A

Reuse the scan flow — skip the camera

Add a "Enter Manually" button alongside "New Scan". It creates a blank card group and drops you into the existing review stage — the same ParsedCardEditor + Met As / Occasion / Date panel you already use after scanning. No images required.

Pros

  • Zero new UI components — reuses everything
  • Identical editing experience to scanned cards
  • Person + Card stored consistently in DB

Cons

  • Backend needs to allow Card with no images
  • Scan flow UX slightly repurposed
B

New dedicated "Manual Entry" page

A separate route /manual with a purpose-built form: name fields, language selector, company, phone, email, date, occasion, Met As. Submit creates Person + contacts via individual API calls. No Card is created.

Pros

  • Clean, focused UX for manual entry
  • No backend changes needed

Cons

  • Lots of new frontend code
  • Duplicates ParsedCardEditor logic
  • Met As / Occasion have nowhere to live (those are card-level fields)
C

New Person button → PersonDetailPage inline editing

A "+ New Person" button on the Persons tab creates an empty Person immediately and redirects to PersonDetailPage. The user then fills in names, contacts, positions using the existing inline edit controls.

Pros

  • Minimal new code
  • No backend changes

Cons

  • No way to set Met As, Occasion, or Date — those are card-level
  • Fragmented UX: many separate inline edits
  • Poor experience for initial data entry

⭐ Recommended: Option A — reuses all existing components and keeps Met As / Occasion / Date naturally in place as card-level fields, even for image-free entries.
